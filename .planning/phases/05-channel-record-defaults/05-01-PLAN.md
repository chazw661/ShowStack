    Exact field definitions (literal — copy verbatim):

    ```python
    default_record = models.BooleanField(
        default=True,
        help_text="If True, new multitrack sessions pre-check (enable) tracks created from this channel. POL-01.",
    )
    default_record_color = models.CharField(
        max_length=7,
        blank=True,
        default='',
        help_text="Optional hex seed color (#RRGGBB) for tracks created from this channel. Empty = no seed; per-track override always wins. POL-02.",
    )
    ```

    Rationale for defaults:
    - `default_record = True`: backwards-compatible — every existing channel becomes a default-on track, matching today's behaviour where every picker-added track is enabled by default. Engineers opt OUT obvious-don't-record channels (e.g. talkback, monitor sends) rather than opt IN every gig.
    - `default_record_color = ''` (empty string, not None): matches MultitrackTrack.color_override's `blank=True, default=''` contract so Plan 03's seed-copy is a one-liner with no None-handling. `max_length=7` matches color_override exactly so the values are interchangeable (#RRGGBB).

    DO NOT:
    - Add the fields to ConsoleImport (it's an audit-snapshot model, not a channel).
    - Touch the existing `color` (Yamaha palette NAME) field — that field stays untouched and continues to serve Rivage/Nuendo exporters and CSV import.
    - Add `choices=` to default_record_color — Plan 03 will copy this verbatim into MultitrackTrack.color_override, which accepts any 7-char hex; constraining to palette swatches would break engineers who paint with non-palette hexes via the swatch picker's custom-color path.
    - Add a validator on default_record_color at the model layer. Hex-format validation lives at the form layer (Plan 02) and the picker-view layer (Plan 03's add_tracks defence-in-depth), matching the same defence-in-depth pattern Phase 1 used for color_override (no model validator; _HEX_COLOR_RE at the AJAX boundary).
