required by `build_nlpr()` and the view layer. This is a **Charlie-owned
prerequisite**: the file must be saved by a human running Nuendo Live 3 on

inside a CLI session — the application's save-to-disk routine is the

`OwnInputBus`, `SendFolder`, ~150 lines per track) that the exporter then
deep-copies into every output track.

but no end-to-end download is meaningful until the real template lands.
Plan 05 (view + URL) and the manual round-trip HUMAN-UAT both depend on
this artifact.

  <how-to-verify>
    **Step 1 — Generate the fixture on Charlie's Windows machine running
    Nuendo Live 3:**

    1. Open Nuendo Live 3 on Windows.
    2. File → New Project. Use the **default empty project template** (no
       audio interface routings, no markers, no signature changes, no
       tempo changes). Stock 120 BPM, 48 kHz sample rate.
    3. Confirm there is exactly ONE audio track in the project (Nuendo's
       default empty session ships with this). Its name should already be
       `Audio 01`.
    4. **Do not rename, recolor, route, or otherwise modify the track.**
       If Nuendo defaulted the track to red (Farb=0) — leave it. Plan 02's
       `_apply_farb` handles both "Farb present in seed → mutate" and
       "Farb absent → SubElement add" cases (RESEARCH A5).
    5. File → Save As → name the file `nuendo_live_3_template.nlpr` and
       save it to a scratch location on Charlie's machine.
    6. Quit Nuendo Live 3 cleanly (avoids the auto-save side files mixing
       in).

    **Step 2 — Verify the saved file before committing:**

    Open the `.nlpr` in a text editor (it's XML). Confirm:
    - First line is an XML declaration with `encoding='UTF-8'` (or
      `utf-8`) — case-insensitive; either works.
    - The file contains `Platform="WIN64"` inside the
      `<obj class="MApplicationVersion">` (or similar — exact element name
      per spec) block. This is the Windows-saved marker.
    - There is exactly ONE `<obj class="MFolderTrack">` whose inner
      `<string name="Name" value="Audio" wide="true"/>` reads `Audio`
      (and a second MFolderTrack named `Input/Output Channels` which is
      fine — exporter's XPath specifically targets `Audio`).
    - The Audio folder contains exactly ONE
      `<obj class="MAudioTrackEvent">` (the seed Plan 02 deep-copies).

    A bash check after copying to the repo (Charlie or Claude can run
    this once the file is in place):
    ```bash
    grep -c '<obj class="MAudioTrackEvent"' planner/data/multitrack/nuendo_live_3_template.nlpr
    # → should be 1
    grep -c 'name="Audio"' planner/data/multitrack/nuendo_live_3_template.nlpr
    # → should be ≥ 1
    grep -c 'Platform="WIN64"\|Platform=\x27WIN64\x27' planner/data/multitrack/nuendo_live_3_template.nlpr
    # → should be 1
    ```

    **Step 3 — Copy into the repo:**

    Create the directory if needed (Comm Config + CSV fixtures use the
    `planner/data/<module>/` convention; this is a new subdirectory):
    ```bash
    mkdir -p planner/data/multitrack/
    cp /path/to/Charlies/nuendo_live_3_template.nlpr \
       planner/data/multitrack/nuendo_live_3_template.nlpr
    ```
