# Phase 4: Nuendo Live Export — Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Engineer can click "Export to Nuendo Live (.nlpr)" on a `MultitrackSession` and
download a Nuendo Live 3 project file that opens cleanly on Windows + Nuendo
Live 3 with one track per enabled `MultitrackTrack`, correct names (outer
`Name` and inner `DeviceAttributes → Name → String`), correct `Farb` palette
indices for tracks whose color resolves to a Yamaha palette name, and `Farb`
omitted for tracks without a recognized palette color. Every `ID` and
`RuntimeID` within the document is unique.

Implementation is **template-injection via `lxml`** — load a bundled empty
`.nlpr`, locate the `Audio` MFolderTrack's single seed track, deep-copy it
once per export-time `MultitrackTrack`, mutate names / Channel ID / Farb,
regenerate all IDs, append back as siblings inside the `<list>`. Everything
outside the Audio folder (`Devices` block, `Input/Output Channels`,
`WindowLayouts`, marker / tempo / signature tracks, `UColorSet`, transport
state) passes through untouched.

**In scope:**
- `planner/utils/nuendo_live_export.py` — pure `build_nlpr(session) → bytes`,
  no DB writes.
- `multitrack_export_nlpr` view + URL route, mirroring the Phase 1 Reaper view
  pattern (`planner/views.py:6876`).
- Per-track `MultitrackTrack.resolved_yamaha_name` property — new color helper
  (does NOT mutate the existing `resolved_color` contract).
- `YAMAHA_TO_FARB` constant inside `nuendo_live_export.py` (8 entries; Off and
  White both map to "omit Farb").
- "Export to Nuendo Live (.nlpr)" button in `editor.html` toolbar alongside
  the two existing Reaper buttons.
- Enable the `target_daw='nuendo_live'` choice in the new-session form
  dropdown (the model choice already exists; the UI gate is the only change).
- Bundled fixture at `planner/data/multitrack/nuendo_live_3_template.nlpr`
  hand-generated by Charlie on Windows + Nuendo Live 3 before plan execution
  can finish.
- `lxml` added to `requirements.txt`.

**Not in this phase:**
- No migrations. Phase 4 ships zero new persistent fields and zero
  `ALTER TABLE` against existing tables.
- No change to `MultitrackTrack.resolved_color` — the hex contract stays
  exactly as Phase 1 shipped, so the Reaper exporter is byte-stable across
  this phase.
- Rivage-specific Farb table — Phase 2 already collapses Rivage colors to the
  10-name CL/QL palette via `YAMAHA_COLOR_CHOICES`, so one table covers all
  three console families. (Closes STATE.md open question #5.)
- Pro Tools `.txt` / AAF (deferred to v2.1, REQUIREMENTS.md PT-01).
- Nuendo Live "track template" / merge-into-existing-project variant — spec
  explicitly states `.nlpr` is the only target format.
- Snapshot byte-for-byte regression tests, structural XML diff tests, or
  automated round-trip-via-real-Nuendo tests. Only one automated assertion
  ships: ID/RuntimeID uniqueness (verifies NLP-06 directly).

## Implementation Decisions

### Template Fixture Lifecycle

- **D-01:** Bundled fixture lives at
  `planner/data/multitrack/nuendo_live_3_template.nlpr`. Mirrors the existing
  convention used by `planner/data/csv_fixtures/` (Phase 2) and
  `planner/data/comm_config/pouchdb_factory/` (Comm Config) for opaque
  vendor-binary reference artifacts.
- **D-02:** Generation contract for the fixture (Charlie hand-generates on
  Nuendo Live 3 before plan execution finishes):
  - Exactly **1 default audio track** inside the `Audio` MFolderTrack, named
    `"Audio 01"` (spec §"Nuendo Live (.nlpr)" template-injection step 1).
  - Empty session settings — stock 120 BPM, 48 kHz sample rate, no markers,
    no signature changes, no tempo changes.
  - Save platform: **Mac is acceptable** (amended 2026-05-13 — Charlie has no
    Windows access). The `Application Version` block will report a Mac
    platform string instead of `Platform="WIN64"`; Nuendo Live's loader does
    not validate this field, so end users on Windows can still open the
    exported `.nlpr`. The spec was reverse-engineered from Windows-saved
    files, so subtle structural differences in Mac-saved XML are
    *theoretically possible but unverified* — the HUMAN-UAT round trip
    (open the exporter's output back in Nuendo Live 3) is the catch-all.
  - No audio interface routings beyond Nuendo's defaults — the `Devices`
    block and `Input/Output Channels` folder pass through untouched.
- **D-03:** Runtime behavior when the fixture is missing, empty, or doesn't
  parse to exactly 1 audio track inside the `Audio` MFolderTrack: render
  `editor.html` with an `export_error` string, no download attempt. Matches
  the Phase 1 no-enabled-tracks pattern at `planner/views.py:6900-6912`.
  Suggested banner copy: *"Nuendo Live export is unavailable on this server
  — bundled template missing or malformed. Contact support."* — planner /
  executor finalizes the exact copy.

### Color → Farb Resolution

- **D-04 (amended 2026-05-14 per HUMAN-UAT feedback):** Add a new property
  `MultitrackTrack.resolved_yamaha_name` (Phase 4 helper, separate from
  `resolved_color`). Resolution order:
  1. If `color_override` is set AND its hex value exists as a value in
     `YAMAHA_TO_HEX` (`planner/utils/reaper_export.py:26-37`), return the
     matching palette name. (Reverse-lookup; only 8 of the 10 Yamaha names
     have hex values — `Off` and `White` map to `None` in `YAMAHA_TO_HEX` and
     are not reachable via the override path.)
  2. Else, return `None`.

  Original D-04 had a step 2 that fell back to `resolved_source.color`
  (the channel-level Yamaha palette name from Phase 2 D-07). HUMAN-UAT
  on 2026-05-14 surfaced a bad UX consequence: the editor's color
  swatch consults `color_override` only, but the Nuendo exporter
  pulled Rivage-CSV-imported channel colors through to Farb. Engineers
  saw tracks render in Nuendo with colors the editor never previewed
  (e.g. Pod 1 / Pod 2 rendering Green, Vid 1L rendering Pink). The
  editor must be the single source of truth for "what color
  exports". Resolution is now symmetric with `resolved_color`:
  override-only. Engineers who want a channel-level color in Nuendo
  must explicitly set it via the editor's color picker. Leaves
  `resolved_color` (hex) untouched. The Reaper exporter is unaffected
  by Phase 4 — its byte-stable Phase 1 contract holds.
- **D-05:** When `color_override` is a hex that does NOT match any value in
  `YAMAHA_TO_HEX`, the exporter emits **no `Farb` element** for that track,
  so Nuendo Live renders the default appearance. Same outcome as no color
  set. Rationale: engineers who painted a precise hex shade (e.g.
  `#A1B2C3`) signaled "I want this exact color" — Nuendo can't reproduce
  arbitrary hex within the 16-color palette, so we don't guess.
- **D-06:** Single `YAMAHA_TO_FARB` table covers CL/QL and Rivage consoles.
  Phase 2 D-07 (amended L-05) already constrains channel-level color storage
  to `YAMAHA_COLOR_CHOICES` (the 10-name CL/QL palette) regardless of the
  source console family — Rivage colors are already collapsed at import
  time. **Closes STATE.md open question #5** ("Rivage→Farb mapping table is
  not yet defined in the spec").
- **D-07:** `YAMAHA_TO_FARB` constant lives in
  `planner/utils/nuendo_live_export.py`, mirroring `YAMAHA_TO_HEX` in
  `planner/utils/reaper_export.py`. Each exporter module owns its
  target-format color table. Locked entries (from spec §"Yamaha → Nuendo
  color mapping"):

  | Yamaha name | Farb | Notes               |
  |-------------|------|---------------------|
  | `Off`       | omit | no color override   |
  | `Red`       | 0    |                     |
  | `Orange`    | 1    |                     |
  | `Yellow`    | 2    |                     |
  | `Green`     | 5    |                     |
  | `Sky Blue`  | 8    | maps to cyan        |
  | `Blue`      | 10   |                     |
  | `Purple`    | 12   |                     |
  | `Pink`      | 14   |                     |
  | `White`     | omit | use Nuendo default  |

### ID & RuntimeID Generation

- **D-08:** Deterministic sequential allocation. At module load (or first
  export call — planner's discretion), scan the bundled template once with
  `lxml` and find the maximum existing integer value across every `<obj
  ID="...">` attribute and every `<int name="RuntimeID" value="..."/>`
  child. Allocate fresh IDs sequentially starting from `max + 1000` per
  export. Same input session → identical output IDs across runs. No
  collision with template-internal IDs. Snapshot regression tests (if any
  are added later) become trivial.
- **D-09:** Test coverage = ONE automated assertion. Parse the generated
  `.nlpr` bytes back with `lxml`, collect every `ID` attribute and every
  `RuntimeID` value into a list, assert `len(set(values)) == len(values)`.
  Directly verifies success criterion 5 / NLP-06. No snapshot test, no
  structural XML diff, no automated round-trip-via-real-Nuendo. Manual
  open-in-Nuendo-Live-3 round-trip per spec §"Round-trip test plan" stays
  on Charlie (HUMAN-UAT item).
- **D-10:** **All IDs replaced for every track**, including track 1. The
  bundled template's existing track-level IDs are seed data only — never
  reused in the output. Every exported `MAudioTrackEvent` / `MListNode` /
  inner `MAudioTrack` / `RuntimeID` comes from the sequential allocator.
  Single source-of-truth invariant: every track row in the output was
  generated identically. Easier to reason about than "track 1 reuses
  template IDs, 2..N are fresh."

### Export Button Visibility & target_daw Semantics

  gate is the only change. New sessions still default to `target_daw='reaper'`
  — minimal behavior change for existing beta users. Planner identifies the

  (likely a JS hint or a form `disabled` attribute) and removes it.
- **D-13:** Nuendo Live button reuses the existing toolbar button styles

  Reaper "Track Template" button uses `mts-btn mts-btn-secondary` as the

- **D-14:** Single `.nlpr` button — no "Nuendo Live track template" variant.
  Per spec §"Nuendo Live (.nlpr)": *"No separate track-archive format exists
  in Nuendo Live (unlike full Nuendo). `.nlpr` is the only target."* The
  toolbar's three-button row is: `.RPP` (primary success), `.RTrackTemplate`
  (secondary), `.nlpr` (primary success).

The planner / executor decides these — defaults to existing project patterns:

- **Exact wave structure for plan execution.** Suggested mapping (planner
  refines):
  - Wave 1: `lxml` dependency add + `resolved_yamaha_name` property +
    `YAMAHA_TO_FARB` constant + pure exporter `build_nlpr(session) → bytes`
    + ID-uniqueness unit test against a Python-generated fake template.
  - Wave 1 blocker: Charlie generates and commits
    `planner/data/multitrack/nuendo_live_3_template.nlpr` before Wave 2
    can finish — the real template is needed for any end-to-end assertion.
  - Wave 2: view + URL route + Content-Disposition headers + no-enabled-
    tracks guard + missing-fixture guard.
  - Wave 3: enable `nuendo_live` in new-session form + add the third
    toolbar button in `editor.html`.
- **Module-load vs first-export fixture parse timing.** Either is fine;
  planner picks based on which is cleaner to test. Module-load gives
  startup safety; first-export defers cost.
- **Whether to add a Django system check** that verifies the fixture parses
  at deploy time. Not required (D-03 handles runtime gracefully) but
  defensible polish if it's <10 lines.
- **`lxml` version pin.** Pick a recent stable (≥4.9.x) and pin to `~=`.
- **CRLF / encoding preservation strategy.** Spec notes some real-world
  `.nlpr` files use CRLF, others bare CR. `lxml.etree.tostring` with
  `xml_declaration=True, encoding='utf-8'` produces LF — verify Nuendo Live
  3 accepts LF on import (real files vary, so the parser appears lenient).
  If LF doesn't work, planner explicitly writes CRLF.
- **Permission gate on the export view.** Phase 1's
  `multitrack_export_rpp` uses `@staff_member_required`; the Phase 1 code
  review surfaced CR-01/CR-02 about server-side authorization gaps. Phase 4
  matches whatever Phase 1 ships post-fix — if Phase 1 has been retightened
  to role-based gating (superuser / premium owner / editor), Phase 4 uses
  the same gate. Planner verifies the current Phase 1 state when wiring the
  view.
- **Filename for the download.** `<_safe_filename(session.name)>.nlpr`,
  reusing the helper at `planner/views.py:6856`.
- **Whether to expose `build_nlpr` from a public module path** for future
  unit tests outside `planner.tests` (e.g. mirror Phase 1's approach where
  `planner/utils/reaper_export.py` exports `build_rpp` + `build_rtracktemplate`
  at module top level).
- **Whether `MultitrackTrack.resolved_yamaha_name` is a `@property`** or a
  helper function (`planner/utils/nuendo_live_export.py:resolve_yamaha_name(track)`).
  Property keeps the call site clean and parallels `resolved_label` /
  `resolved_color` / `resolved_dante_number`. Function keeps the model
  thin. Planner picks; recommendation is property for consistency.

## Specific Ideas

- **Engineer mental model carried through the discussion:** *"Click export,
  get a `.nlpr` that opens in Nuendo Live with my track names and the right
  colors. Anything fancier — perfect color matching, custom palettes — I'll
  fix inside Nuendo if I care."* This anchored D-05 (non-palette hex → omit
  Farb rather than guess at a nearest neighbor).
- **Beta-tester safety constraint applies even though no migrations ship.**
  The new exporter must never raise an uncaught exception on real session
  data. D-03's render-editor-with-banner approach for missing fixtures, and
  the planner's responsibility to keep the exporter pure (no DB writes), are
  both downstream of this.
- **Test budget — one assertion.** D-09 captures the user's explicit choice
  to ship ONE automated test rather than three. Rationale: NLP-06 is the
  only fully-machine-verifiable criterion. NLP-01..05 ultimately require
  opening the file in Nuendo Live 3, which is a HUMAN-UAT item. Snapshot
  tests against a generated `.nlpr` would lock the exporter to a specific
  output byte sequence that real Nuendo Live may not care about, so the
  test cost would exceed the regression value. **If during planning a
  cheap structural test surfaces** (e.g. "every output `<obj
  class='MAudioTrackEvent'>` count == `enabled_track_count`") **the planner
  may add it under Claude's Discretion** — but the floor is exactly one
  test, and that test is the ID/RuntimeID uniqueness check.
- **No `target_daw` gate on the export buttons.** Today, both Reaper buttons
  show even when `target_daw='nuendo_live'`. After Phase 4, all three
  buttons show regardless of `target_daw`. The field's semantic meaning
  becomes "default DAW for new sessions / template signaling for
  `MultitrackTemplate`" rather than "what export buttons appear on this
  session." Don't refactor that field away — Phase 3 templates carry
  `target_daw` for legitimate reasons.
- **The "save on Windows + Nuendo Live 3" requirement (D-02)** is real and
  blocks plan execution. Planner should call this out explicitly in the
  plan as a Charlie-owned prerequisite, alongside the lxml dependency add.
  Suggested wave structure: parser/exporter logic + uniqueness test against
  a Python-generated fake template can land in Wave 1 without the real
  fixture; the real fixture is required for any end-to-end view test and
  for the manual round-trip HUMAN-UAT item.
