<objective>
Atomically remove the three Phase 1 belt-and-suspenders gates that
disable `nuendo_live` in the new-session form. Per CONTEXT.md D-12 and
RESEARCH §"Pitfall 6", the gate lives in **three** places, and removing
only one or two leaves the form half-broken in subtle and confusing ways:
- Remove gate 1 only → user can pick it visually, but server-side
  `clean_target_daw` raises ValidationError.
- Remove gate 2 only → choices-restriction strips `nuendo_live` from
  `self.fields['target_daw'].choices` so the value can't survive form
  binding ("Select a valid choice").
- Remove gate 3 only → an inert disabled `<input type="radio" disabled>`
  still renders alongside the dynamic radios.

This plan ships ALL THREE removals in one commit so the choice flips
from "disabled" to "fully enabled" atomically.

This plan has **no `files_modified` overlap with Plans 02 / 04 / 05**, so
it runs in Wave 2 in parallel with them (`depends_on: []` because gate
removal doesn't need the exporter, the view, or the test to land first;
it only needs `MultitrackSession.TARGET_DAW_CHOICES` from Phase 1, which
already includes `nuendo_live`).

Purpose: Until the gate is removed, engineers can't pick "Nuendo Live"
as the target DAW on a new session. After this plan lands, they can.
NLP-01's full UX is partially gated on this — the export button (Plan 07)
shows for any session, but the new-session form's intent capture is
gated here.

Output:
- `planner/forms.py` loses the choices-restriction block AND the
  `clean_target_daw` method.
- `planner/templates/planner/multitrack/new_session.html` loses the
  static disabled-radio block.
- Zero behavior change for sessions that already exist (their stored
  `target_daw` value is preserved; this only affects new sessions).
- Zero migrations (no model change).
</objective>
