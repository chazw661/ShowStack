**HTML structure (8 components from UI-SPEC):**

1. **Alert banners** (top of nhm-root, hidden when no alerts):
   - Container `#nhm-alerts`
   - Each alert: red dot + device name + "offline" + timestamp + "Dismiss alert" button
   - Populated from `active_alerts` context on initial load
   - New alerts inserted by JS when SSE sends OFFLINE event
   - Dismiss removes banner with opacity+max-height transition

2. **Domain rollup bar** (below alerts):
   - `#nhm-rollup` horizontal flex row, 40px height
   - Three pills: "LA Network: {online}/{total}", "Dante: 0/0", "Switches: 0/0"
   - LA Network pill color based on status. Dante and Switches pills grey (--text-dim) as Phase 1 placeholders
   - Populated from `domain_counts` context on initial load
   - Updated by JS on STATUS_SNAPSHOT SSE events

3. **Scan controls** (header area, right side):
   - Page title "Network Health Monitor" (16px/700, --accent-blue)
   - NIC `<select>` populated from `nics` context
   - "Start Scan" button (--accent-blue bg, 11px/700 uppercase)
   - Scan results area (hidden until scan completes): checkboxes + IP + latency rows
   - "Add to Monitor" button (activates when checkboxes selected)
   - Domain `<select>` for assigning domain to selected devices (la_network/dante/switch/unknown)
   - "Scanning..." disabled state with CSS spinner

4. **Domain sections** (main content area, one per domain):
   - Three sections: LA Network, Dante, Switches
   - Each has header: domain name + device count badge + collapse toggle (unicode triangle)
   - Card grid: CSS grid, `repeat(auto-fill, minmax(260px, 1fr))`, 8px gap
   - Collapse state in localStorage key `nhm_domain_{slug}_collapsed`
   - Populated from `devices` context grouped by domain

5. **Device cards** (inside domain sections):
   - Collapsed: `[status dot] [device name] [latency or "---"]`
   - Click expands: IP address (mono), last seen (mono dim), failure count, "Remove from monitor" button
   - Status dot class mapped from device.status(): online/flapping/offline/unknown
   - Expand state in localStorage key `nhm_card_{device_id}_expanded`
   - "Remove from monitor" triggers confirmation dialog then POST to remove endpoint

6. **Session history timeline** (below domain sections):
   - Toggle header "Session History" with collapse triangle, collapsed by default
   - Event rows: `[timestamp mono dim] [status dot] [device name] [event description]`
   - Event type labels: ONLINE="came online", OFFLINE="went offline", SCAN_STARTED="Network scan started", MONITOR_STARTED="Monitor started"
   - Max 20 visible rows, scroll within container
   - Populated from `recent_events` context on initial load
   - New events prepended by JS from SSE stream

7. **"Not on show network" empty state**:
   - Shown when `has_project` is true but `devices` is empty and `monitor_running` is false
   - Heading: "Monitor is not running. Start it with: python manage.py run_monitor"
   - When `has_project` is true, `devices` is empty, and `monitor_running` is true:
   - Heading: "Not connected to show network"
   - Body: "No devices are being monitored. Run a scan to discover devices on your network."
   - CTA: scroll to scan controls

8. **SSE connection indicator** (fixed bottom-right):
   - Connected: hidden (silence = healthy)
   - "Reconnecting..." amber text when EventSource errors
   - "Live updates paused --- reload page" amber banner after 5 retries

**JavaScript section must include:**

1. **CSRF token extraction** (Django pattern):
   ```javascript
   function getCookie(name) {
       // Standard Django CSRF cookie extraction
   }
   const csrfToken = getCookie('csrftoken');
   ```

2. **EventSource connection** with reconnect (per RESEARCH.md Pattern 5):
   - Connect to `/audiopatch/network-monitor/stream/`
   - `onmessage`: parse JSON, route by event.type to handler functions
   - `onerror`: close, increment retries, reconnect with exponential backoff (max 10s), show "Reconnecting..."
   - `onopen`: reset retries, hide reconnecting indicator
   - Max 5 retries before showing "Live updates paused"

3. **Event handlers:**
   - `handlePollUpdate(event)` for STATUS_SNAPSHOT: update all device dots and rollup bar counts
   - `handleOffline(event)`: set device dot to offline, add alert banner, prepend to timeline
   - `handleOnline(event)`: set device dot to online, prepend to timeline

4. **DOM update functions:**
   - `updateStatusDot(deviceId, status, latencyMs)`: find `#nhm-device-{id}`, update dot class and latency text
   - `updateRollupBar()`: recount online/total per domain from DOM state, update pill text and color
   - `showAlertBanner(deviceName, deviceId, occurredAt)`: insert alert banner HTML at top of `#nhm-alerts`
   - `dismissAlert(deviceId)`: animate out banner row
   - `prependTimelineEvent(eventData)`: add row to `#nhm-timeline-list`

5. **Scan flow:**
   - Select NIC from dropdown
   - Click "Start Scan" -> POST to `/audiopatch/network-monitor/scan/` with `{interface, subnet}` from selected NIC
   - Show "Scanning..." spinner state
   - On response: render checkbox list of discovered devices with IP and latency
   - Mark already-monitored devices as checked+disabled
   - "Add to Monitor" -> POST to `/audiopatch/network-monitor/devices/add/` with selected devices
   - On success: reload page to pick up new devices

6. **Device removal:**
   - "Remove from monitor" button shows confirm dialog using copywriting contract text
   - On confirm: POST to `/audiopatch/network-monitor/devices/{id}/remove/`
   - On success: remove card from DOM, update rollup counts

7. **localStorage state persistence:**
   - Domain section collapse: read on load, toggle on click, save on change
   - Device card expand: read on load, toggle on click, save on change
   - Keys: `nhm_domain_{slug}_collapsed`, `nhm_card_{device_id}_expanded`

8. **Accessibility:**
   - `aria-label` on every status dot: `"{device_name}: {status}"`
   - Alert banners: `role="alert"`
   - All buttons have visible focus rings via CSS
   - Scan results checkboxes have labels

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-01 | Tampering | DOM manipulation | accept | Client-side state is non-authoritative; server is source of truth. Dismissing alert in DOM does not change DB state. |

| T-03-03 | Spoofing | CSRF on POST endpoints | mitigate | All fetch POST calls include X-CSRFToken header from Django csrftoken cookie |
